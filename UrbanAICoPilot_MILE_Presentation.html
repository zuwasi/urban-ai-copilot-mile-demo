Why This Paper

MILE was the only executable candidate

The selection criterion was technical: state variables, objective terms, rollout logic, and measurable outputs that could survive a Wolfram implementation.

Selection Logic

  • AV2.0 was architecture commentary with limited equation-level structure for a CAS-driven reconstruction workflow.
  • MILE exposed the right primitives: recurrent memory h_t, stochastic state s_t, posterior/prior moments, and a diagonal Gaussian KL.
  • MILE also exposed deployable measurements: frequency, ablations, recurrent/reset-state tradeoffs, and scenario-dependent behavior we could re-simulate.

Product Translation

Research model versus product wrapper

The technical design rule was simple: do not hide product logic inside the latent model. Keep the learned layer explicit, then attach deterministic interfaces around it.

“A mapless L2+/L3 urban driving system that uses the MILE-style world model as the performance layer, but wraps it with a trajectory interface, uncertainty-aware gating, and an independent safety supervisor.”

This sentence is taken directly from the published notebook’s research-context section.

Architecture We Explained

Latent state update layer

PosteriorMoments, PriorMoments, and UpdateDeterministicState implement the recurrent latent transition logic.

Control and trajectory layer

PredictAction and PredictTrajectory map latent state into acceleration, steering, waypoints, and speed profiles.

Post-policy supervisory layer

SafetySupervisor applies stop-line clipping and obstacle buffering after trajectory generation, preserving separability for audit and explanation.

Imagination Mode

Observe versus imagine branch behavior

Observation dropout is treated as a geometric prior-only unroll mechanism. In the implementation, imagination steps explicitly widen uncertainty while preserving rollout continuity.

Safety Layer

SafetySupervisor as a post-policy module

The autonomy-specific extension is not vague “guardrails.” It is a deterministic post-policy transform applied after trajectory proposal and before metric accounting.

Implemented intervention rules

  • Red-light ruleIf SignalState == "Red" and the path crosses the stop line, y-coordinates are clipped and the speed profile is collapsed to minimum-risk speed.
  • Obstacle buffer ruleIf MinimumDistance < SafetyMarginMeters, the path shifts laterally and speed is scaled by 0.55.
  • Explicit metric impactRisk improves, intervention rate rises, and reward can fall because the supervisor is intentionally conservative.

Regression Case Study

The BEV bug and how we closed it

After publishing the notebook, the BEV visualization exposed a real implementation bug: the model loaded, but the decoded grid had the wrong shape for ArrayPlot.

Root Cause

  • Observed failureArrayPlot::mat reported that the first argument was not a list of lists when rendering bev["CombinedGrid"].
  • Underlying defectDecodeBevState accumulated obstacle contributions as vectors instead of scalars, so CombinedGrid became {48,48,2} rather than {48,48}.
  • Direct fixThe obstacle field was rewritten to sum scalar obstacle influence per cell using each obstacle's Severity and Position.
Broken shape {48,48,2}
Fixed shape {48,48}
ArrayPlot returns Graphics again

Why the audit matters

The important engineering lesson is not just that we fixed the bug. We promoted it into a regression test so the same failure mode cannot silently reappear.

  • ValidationChecks now asserts that Dimensions[bevState["CombinedGrid"]] equals the configured square BEV size.
  • proof_audit.wls now includes a dedicated audit: “BEV decode returns a 2D matrix for ArrayPlot”.
  • Net effectThe deck, notebook, and audit now all agree on the same invariant: BEV decode must return a matrix.
